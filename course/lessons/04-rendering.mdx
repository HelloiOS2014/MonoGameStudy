## Goal

Draw sprites in 2D and understand why `SpriteBatch` batching matters for performance.

## Run

```bash
env E02_SMOKE_TOGGLE_AFTER_FRAMES=180 E02_SMOKE_EXIT_AFTER_FRAMES=340 dotnet run --project experiments/e02-2d-rendering/E02Rendering.csproj --no-restore
```

## Observe

The window shows 1000 sprites and overlay text. The smoke run starts in `mode=Batched`, then logs:

```text
Update: render mode changed to Unbatched.
Smoke: exit.
```

## Expected Visual State

Window shows 1000 sprites plus overlay text; smoke toggles from batched to unbatched mode.

## Walkthrough

MonoGame's common 2D drawing path is `SpriteBatch`: call `Begin`, queue texture and text draws, then call `End` to flush work to the GPU.

The experiment draws 1000 sprites in two modes. `Batched` wraps all sprite draws in one `Begin`/`End` pair. `Unbatched` intentionally uses one pair per sprite to make broken batching visible. The overlay uses `SpriteBatch.DrawString` and a loaded `SpriteFont` to print teaching signals, not benchmark-grade numbers.

`RenderModeState` keeps the mode toggle outside rendering calls, so the draw path can stay focused on `SpriteBatch` usage. The smoke variables make the GUI lesson verifiable by logging the same mode change the learner sees in the overlay.

## Common Failures

- Missing `Status.spritefont` or built content usually means restore/build did not finish cleanly.
- `Content.Load<SpriteFont>("Status")` uses the logical asset name, not the file extension.
- Very early FPS samples can include startup/content-load cost and should not be treated as stable measurements.

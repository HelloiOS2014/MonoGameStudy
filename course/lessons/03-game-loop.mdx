## Goal

Understand fixed vs variable timestep in MonoGame and why the repo uses smoke variables to make a GUI loop testable.

## Run

```bash
env E01_SMOKE_TOGGLE_AFTER_FRAMES=30 E01_SMOKE_EXIT_AFTER_FRAMES=90 dotnet run --project experiments/e01-game-loop/E01GameLoop.csproj --no-restore
```

## Observe

The smoke run opens briefly, toggles once, and exits. Stdout includes:

```text
Update: timestep mode changed to Variable.
Smoke: exit.
```

If you run manually, press `F1` and watch the window title switch between `Fixed 60 Hz` and `Variable`.

## Expected Visual State

e01 window starts fixed timestep, smoke toggles to variable timestep, stdout includes `Update: timestep mode changed to Variable.`.

## Key Files

- `experiments/e01-game-loop/Game1.cs`
- `experiments/e01-game-loop/GameLoopState.cs`
- `experiments/e01-game-loop/LoopSmokeSettings.cs`
- `experiments/e01-game-loop.Tests/Program.cs`

## Walkthrough

MonoGame exposes timestep behavior through `IsFixedTimeStep`. When it is true, this experiment targets 60 updates per second. When it is false, MonoGame runs in variable timestep mode.

`GameLoopState` keeps the mode outside `Game1` so behavior can be tested without graphics. `LoopSmokeSettings` reads `E01_SMOKE_TOGGLE_AFTER_FRAMES` and `E01_SMOKE_EXIT_AFTER_FRAMES`, then `Update` treats the configured frame as if the toggle was pressed. That is more reliable on macOS than depending on synthetic keyboard events for a GUI app.

## Common Failures

- Synthetic key events can be unreliable on macOS, so automated smoke uses environment variables.
- `E01_SMOKE_EXIT_AFTER_FRAMES` must be greater than `E01_SMOKE_TOGGLE_AFTER_FRAMES`.
- `--no-restore` assumes the project has already restored. Rerun without it once if packages are missing.
